# GCP 무료티어 + Cloudflare Tunnel — 외부 IP 없이 외부 접속이 되는 이유

## 한 줄 요약

외부에서 들어오는(inbound) 길을 여는 대신, **VM이 먼저 바깥으로 나가서(outbound) Cloudflare에 연결해둔 통로**로 트래픽을 받는다. 그래서 VM에 공인 IP(외부 IP)가 없어도 인터넷에서 접속이 된다.

---

## 왜 보통은 외부 IP가 필요한가

일반적인 웹 서버 노출은 이렇게 동작한다.

```
브라우저 → (인터넷) → 서버의 공인 IP:포트 → 앱
```

브라우저가 서버에 **직접 접속**하려면, 서버에 "인터넷 어디서든 찾아올 수 있는 주소" 즉 공인 IP가 있어야 한다. 이게 일종의 **정문 주소**다. 정문이 있으면:

- 그 주소로 누구나 노크할 수 있다 (편함)
- 동시에 아무나 노크할 수 있다 (포트 스캔·공격 대상이 됨)
- GCP에선 이 외부 IP 자체가 과금 대상 (월 $3~4)

핵심 제약: **inbound 연결은 "받는 쪽"에 도달 가능한 주소가 있어야 성립한다.**

---

## 그런데 outbound는 IP가 없어도 된다

여기가 전체 구조의 열쇠다.

VM에 외부 IP가 없어도 **바깥으로 나가는 연결(outbound)** 은 멀쩡히 된다. 왜냐하면:

- VM이 인터넷으로 나갈 때는 클라우드의 NAT/게이트웨이를 통해 나간다
- 나갈 때 "내가 먼저 연결을 걸었다"는 사실 자체가 돌아올 길을 만든다 (집에서 웹사이트 접속할 때 우리 집에 공인 IP가 없어도 응답이 돌아오는 것과 같은 원리)

즉, **"남이 나를 찾아오는 길"(inbound)은 막혀 있어도, "내가 남에게 가는 길"(outbound)은 열려 있다.**

Cloudflare Tunnel은 이 비대칭성을 이용한다.

---

## Cloudflare Tunnel의 발상: 방향을 뒤집는다

VM 안에서 `cloudflared`라는 작은 프로그램을 돌리면, 이놈이 **VM 쪽에서 먼저 Cloudflare 엣지로 연결을 건다(outbound).** 그리고 그 연결을 끊지 않고 계속 유지한다.

이렇게 한번 뚫어둔 통로가 "터널"이다. 이제 흐름은 이렇게 된다.

```
브라우저
   ↓  https://<주소>.trycloudflare.com
Cloudflare 엣지          ← 누구나 접속 가능한 공인 주소는 여기가 가진다
   ↓  (아웃바운드 터널: VM이 미리 연결해둔 통로)
VM 안의 cloudflared
   ↓  http://localhost:포트
앱 (Docker 컨테이너 등)
```

포인트:

- **공인 주소(정문)는 VM이 아니라 Cloudflare가 가진다.** 브라우저는 Cloudflare에 접속하는 거지 VM에 직접 접속하는 게 아니다.
- VM은 그저 Cloudflare 쪽으로 미리 통로를 열어뒀을 뿐. VM에는 정문(외부 IP)이 없다.
- 그래서 외부 IP $0, 게다가 HTTPS도 Cloudflare가 자동으로 붙여준다.

---

## 왜 이게 보안적으로도 더 나은가

기존 방식(외부 IP + 포트 개방)은 VM에 정문을 내고 그 앞에 인터넷 전체를 세워두는 셈이다. 포트 스캔·무차별 접근에 노출된다.

터널 방식은 VM에 들어오는 정문 자체가 없다. VM의 방화벽에서 80/443 같은 inbound 포트를 열 필요조차 없다. 바깥에서 VM으로 직접 꽂을 수 있는 경로가 존재하지 않고, 오직 VM이 자기가 먼저 열어둔 통로로만 트래픽이 들어온다.

> 보안 관점 메모: 공격 표면(attack surface)이 "열린 inbound 포트"에서 "내가 통제하는 outbound 터널 하나"로 줄어든다. 대신 신뢰의 무게가 Cloudflare(중간자)와 cloudflared 인증 토큰으로 옮겨간다.

---

## 그럼 SSH(관리 접속)는 어떻게?

외부 IP가 없으면 SSH로 직접 못 들어온다. 이것도 같은 원리로 우회한다.

- **IAP 터널** — GCP 콘솔의 SSH 버튼이나 `gcloud ... --tunnel-through-iap`로 접속. 구글이 중간에서 터널을 뚫어준다.
- (또는 Cloudflare로 SSH용 터널을 따로 하나 더 팔 수도 있다)

즉 **관리 접속(SSH)은 IAP가, 서비스 접속(웹)은 Cloudflare 터널이** 각각 다른 경로로 해결한다. 둘 다 inbound 정문 없이 동작하므로 외부 IP가 필요 없다.

## 내 시연용 구성 (이 구조의 실제 운용)

- GCP 무료티어 인스턴스(e2-micro) 사용
- 외부 IP 제거 → 평소엔 외부에서 들어올 주소가 없음
- 트래픽 노출은 전적으로 Cloudflare 터널에 위임

```
브라우저 → Cloudflare 엣지 → (아웃바운드 터널) → VM의 cloudflared → localhost:포트 → 앱(Docker 등)
```

### 터널 운용 절차

**평소 (닫힌 상태)**
VM 켜둠 + 외부 IP 제거 + Docker 백그라운드(`-d`)로 앱은 계속 돌고 있음
→ 앱은 `localhost:포트`에 살아있지만, 바깥에서 들어올 주소(터널)가 없어 외부에선 접속 불가능한 공간

**시연할 때 (잠깐 여는 상태)**
SSH 열고 → 터널 한 줄 실행 → 주소 받아 공유 → 끝나면 `Ctrl+C`

```bash
# 주소 줄만 깔끔하게 보기
cloudflared tunnel --url http://localhost:8000 2>&1 | grep --line-buffered trycloudflare

# localhost가 IPv6(::1)로 풀려서 connection refused 나면 127.0.0.1로 강제
cloudflared tunnel --url http://127.0.0.1:8000 2>&1 | grep --line-buffered trycloudflare
```

- quick tunnel은 **켜져 있는 동안만** 살고, 켤 때마다 주소가 새로 바뀐다
- 그래서 미리 공유는 못 하고, **시연 직전에 켜서 그 자리에서 주소 공유**
- 일회성 시연엔 오히려 이게 잘 맞음: 보여줄 때만 열고, 끝나면 흔적 없이 닫히고, 평소엔 외부 노출 0

> 고정 주소가 필요해지면(초대장에 주소 미리 박기 등) → 영구 터널(named tunnel) 필요.
> 단, 영구 터널은 도메인이 Cloudflare 네임서버에 올라가 있어야 함. 회사 도메인(bmap.kr)을 통째로 옮기는 건 운영 리스크가 있으니, 그땐 개인 도메인을 따로 두는 게 안전.

## 핵심 개념 한 번 더

| | inbound (남→나) | outbound (나→남) |
|---|---|---|
| 외부 IP 없을 때 | 막힘 (도달할 주소가 없음) | 됨 (NAT로 나감) |
| 누가 주소를 가지나 | 내가 가져야 함 | 상대가 가지면 됨 |

Cloudflare Tunnel = **"내가 주소를 갖는 inbound" 문제를, "상대(Cloudflare)가 주소를 갖고 내가 outbound로 붙는" 문제로 바꿔치기한 것.**

그래서 외부 IP 없이도, 방화벽 포트 안 열고도, HTTPS까지 공짜로 외부 노출이 된다.
