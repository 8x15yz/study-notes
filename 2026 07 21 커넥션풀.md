# 커넥션 풀 (Connection Pool)

## 필요한 이유

- 어떤 연결(connection)이든 처음 맺을 때 비용이 큼
- DB 연결의 경우 대략 아래 단계를 거침
    1. TCP 연결 수립 (3-way handshake)
    2. 인증 (아이디 / 비밀번호 검증)
    3. 세션 설정 (문자셋, 권한 등)
- 이 과정이 매 요청마다 반복되면 느리고 자원 낭비가 심함
- 그래서 연결을 몇 개 미리 만들어 "풀(pool)"에 담아두고 → 요청이 오면 풀에서 하나 꺼내 씀
- 작업이 끝나면 연결을 닫지 않고 다시 풀에 반납해 재사용함

## 어디서 쓰이나

핵심은 "연결을 맺는 비용이 큰 대상"이면 어디든 적용된다는 점임.

- **HTTP 커넥션 풀**: 같은 서버에 여러 번 요청할 때 TCP 연결을 새로 맺지 않고 재사용함. HTTP Keep-Alive가 이 개념이며, 브라우저나 HTTP 클라이언트 라이브러리(예: 파이썬 requests의 Session, 자바 OkHttp)가 내부적으로 커넥션 풀을 사용함
- **gRPC / 마이크로서비스 간 통신**: 서비스끼리 반복 호출할 때 채널을 풀링해 재사용함
- **메시지 브로커 연결**: MQTT, Kafka, RabbitMQ 같은 브로커에 붙는 연결도 풀로 관리하는 경우가 있음
- **캐시 / 저장소 연결**: Redis 같은 대상도 커넥션 풀 개념을 그대로 사용함
- **스레드 풀 / 오브젝트 풀**: 연결은 아니지만, 비싼 자원을 미리 만들어두고 재사용한다는 같은 아이디어를 공유함. 그래서 "풀(pool)"이라는 단어가 여러 곳에서 등장함

## 정리하면

- 커넥션 풀은 "연결 맺는 비용이 크니 미리 만들어두고 돌려쓰자"는 범용 패턴임
- 대상이 DB면 DB 커넥션 풀, HTTP면 HTTP 커넥션 풀인 것뿐임

## 커넥션 풀은 "연결을 거는 쪽(클라이언트 쪽)"이 가짐

- 커넥션 풀은 항상 "연결을 먼저 거는 쪽"에 위치함

| 연결을 거는 쪽 | 연결받는 대상 | 무슨 커넥션 풀 |
| --- | --- | --- |
| 브라우저 | 웹 서버 | HTTP 커넥션 풀 |
| 백엔드 서버 | DB | DB 커넥션 풀 |
| 백엔드 서버 | 외부 API | HTTP 커넥션 풀 |
| 백엔드 서버 | Redis, MQTT 등 | 각 프로토콜 커넥션 풀 |

### 브라우저 → 백엔드 서버 → DB 예시

- 브라우저가 백엔드에 요청할 때: 브라우저가 "거는 쪽", 백엔드가 "받는 쪽"
- 백엔드가 DB에 쿼리할 때: 백엔드가 "거는 쪽", DB가 "받는 쪽"

⇒ 즉 백엔드 서버도 DB 입장에서는 클라이언트임. 그래서 백엔드가 DB 커넥션 풀을 가짐. "백엔드엔 커넥션 풀이 없다"가 아니라, 백엔드는 자기가 연결을 거는 대상(DB, 다른 API)에 대한 커넥션 풀을 가진다가 정확한 표현임

### HTTP에서 쓰이는 사례

- HTTP는 원래 "요청 하나 보내고 연결 끊기"가 기본이었음. 그러면 매번 TCP 연결을 새로 맺어야 해서 느림
- 그래서 **HTTP Keep-Alive**가 등장함 — 연결을 바로 끊지 않고 잠깐 열어둬 다음 요청에 재사용함
- 이 "열어둔 연결들을 모아 재사용하는 것"이 HTTP 커넥션 풀임. 즉 커넥션 풀은 DB 전용이 아니라 HTTP에서도 그대로 쓰이는 개념임
- 예시
    - 브라우저가 같은 사이트에서 이미지·CSS·JS를 여러 개 받을 때 연결 몇 개를 재사용함
    - 백엔드가 외부 API를 자주 호출할 때 `httpx`나 `requests.Session`이 내부 HTTP 커넥션 풀로 연결을 재사용함

## 커넥션 풀 구현 라이브러리

커넥션 풀은 직접 구현하기보다 라이브러리로 제공되는 경우가 많음.

- **파이썬 / DB**: asyncpg (PostgreSQL 비동기 드라이버, 내장 풀), SQLAlchemy (풀 기능 내장), psycopg (풀 지원)
- **파이썬 / HTTP**: requests(Session), httpx, urllib3
- **자바 / DB**: HikariCP, Apache DBCP, c3p0
- **자바 / HTTP**: OkHttp, Apache HttpClient
