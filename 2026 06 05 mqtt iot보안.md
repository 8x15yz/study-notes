# MQTT와 IoT 보안 정리

> 2026-06-05 / 주제: MQTT 페이로드, 홈캠 데이터 전송 구조, IoT 공격 표면, 유명 해킹 사례

---

## 1. MQTT 페이로드와 대용량 데이터

- MQTT 페이로드는 프로토콜상 **최대 256MB**까지 가능하지만, 실무에선 그렇게 쓰지 않음.
- 핵심: **MQTT로 영상 자체를 보내지 않는다.**
- MQTT는 TCP 위에서 동작하는 **경량 pub/sub 메시징 프로토콜** → 대용량 바이너리 전송엔 비효율적.
- 브로커가 모든 메시지를 거쳐가므로 영상까지 통과시키면 병목·메모리 부담.
- QoS, 재전송, retained 같은 기능이 큰 페이로드엔 부적합.

### 홈캠에서의 역할 분담
- **제어/신호 → MQTT**: "녹화 시작", "움직임 감지", "디바이스 온라인" 등 수십~수백 바이트 메시지.
- **영상 데이터 → 별도 채널**:
  - 실시간 스트리밍: RTSP, WebRTC, HLS
  - 녹화 파일 업로드: HTTP(S)로 클라우드(S3 등)에 직접 업로드

### 실제 흐름 예시
1. 카메라가 움직임 감지 → MQTT로 `{"event":"motion","time":...}` 발행
2. 서버/앱이 알림 수신
3. 영상 보고 싶으면 → WebRTC/RTSP로 카메라 직접 연결 or 클라우드 녹화본 HTTP 다운로드

> 비유: MQTT는 "택배 도착했어요" 알림 문자, 실제 택배(영상)는 다른 트럭으로 옴.

---

## 2. 홈캠 = MQTT + HTTP(+RTSP/WebRTC/HLS)

- "신호는 MQTT, 영상 데이터는 별도 채널"로 기억.

### 중요한 오해 교정: "영상 터는 게 결국 DB 터는 거냐?"
→ **절반만 맞음.** 영상은 보통 **DB에 안 들어감.** DB엔 메타데이터만.

- **DB에 저장되는 것**: 계정/디바이스 ID/토큰, 녹화 이벤트 기록, 영상 파일의 **위치(경로/URL)**
- **영상 실물이 있는 곳**: 클라우드 오브젝트 스토리지(S3 등), IoT 장비 로컬(SD/내장 메모리), 실시간 스트림(저장 안 됨)

> DB는 **열쇠고리**, 실제 영상은 스토리지·장비·스트림에 흩어져 있음.

---

## 3. IoT 영상 관련 공격 표면 (Attack Surface)

1. **DB 침해** — 계정 탈취, 영상 위치 파악, 토큰 탈취. DB엔 영상이 없어도 "영상으로 가는 열쇠"가 다 있음.
2. **스토리지 직접 공격** — 잘못 설정된 public S3 버킷. 실제 홈캠 유출 사고의 가장 흔한 경로.
3. **장비 로컬 탈취** — 펌웨어 취약점, 기본 비밀번호, 텔넷 포트 열림 등으로 카메라 장악 (Mirai 방식).
4. **스트림 가로채기** — RTSP/WebRTC 인증 약하거나 미암호화 시 실시간 영상 가로채기 (예: Insecam류 노출 사이트).
5. **MQTT 브로커 자체** — 인증 없이 열린 브로커면 토픽 구독으로 생활 패턴까지 노출.

> 실무 사고는 DB 해킹보다 **설정 미스(public 버킷, 기본 비번, 열린 포트)**에서 더 많이 발생.

## 4. 유명 IoT 해킹 사례

| 사건 | 연도 | 핵심 |
|------|------|------|
| **Mirai 봇넷** | 2016 | 기본 비번 IoT 기기 수십만 대 감염 → Dyn DDoS로 미 동부 인터넷 마비. IoT = "공격 도구로 징발" |
| **Stuxnet** | 2010 | 이란 핵시설 원심분리기(SCADA/PLC) 정밀 공격. 물리 장비 파괴한 최초 사이버 무기 |
| **카지노 수족관 온도계** | 2017경 | IoT 온도계를 교두보(pivot)로 내부망 진입 → 고객 DB 유출 |
| **Ring 홈캠** | 2019~ | 자격증명 스터핑(비번 재사용)으로 가정용 카메라 탈취. 2차 인증 부재 |
| **Verkada** | 2021 | 클라우드 카메라 업체 침해, 약 15만 대 실시간 영상 접근. 관리자 크리덴셜 노출이 발단 |
| **Jeep Cherokee** | 2015 | 주행 중 차량 원격 조작(브레이크/엔진/핸들) → 140만 대 리콜. 커넥티드카 위험성 증명 |

### "IoT는 중요 정보가 없어서 안 터진다"는 오해
오히려 보안이 약해서 더 잘 터짐. 노려지는 이유:
1. **공격 도구로서의 가치** — 봇넷, DDoS, 채굴, 내부망 침투 발판
2. **물리 세계 영향** — 감시, 도어락 해제, 산업설비 파괴
3. **방어가 약함** — 펌웨어 미업데이트, 기본 비번, 긴 수명, 보안 인력 없는 제조사
4. **프라이버시 정보 자체가 민감** — 집 안 영상, 생활 패턴, 위치

## 5. IoT 보안 입문 실습 아이디어

- **Shodan 탐색** — 인터넷 노출 기기 검색. `port:554`(RTSP), `port:1883`(MQTT) 필터로 감 잡기. (보는 것만, 접속 금지)
- **펌웨어 리버싱** — `binwalk -e firmware.bin`으로 파일시스템 추출 → 하드코딩된 비번/키 찾기. 연습용: **OWASP IoTGoat**
- **MQTT 직접 실습** — 로컬에 Mosquitto 브로커 띄우고 인증 없을 때 토픽 구독이 어떻게 새는지 확인
- **Wireshark IoT 트래픽 분석** — 집 IoT 기기의 평문 트래픽 캡처

### 9개 하위분야가 IoT에서 한 점으로 모임
- 웹 보안 → 클라우드 API, MQTT 브로커 인증
- 암호학 → 기기 인증, 펌웨어 서명, TLS
- 네트워크 보안 → 패킷 분석, RTSP 스니핑
- 취약점 연구 → 펌웨어 리버싱

> 대학원 방향: IoT/임베디드 보안, 하드웨어 보안 연구실. 주제 예) 사이드채널 공격, 시큐어 부트, TEE

## 6. Shodan — "세상에서 가장 무서운 검색엔진"

### 누가, 왜 만들었나
- **John Matherly**가 **2009년** 출시. 2009년 DEFCON 17에서 공개됐고 언론이 "세상에서 가장 무서운 검색엔진"이라 부름.
- 이름 Shodan은 게임 **System Shock**의 악역 AI 캐릭터에서 따옴.
- 원래 목적은 악의적이지 않음 → 연구자·보안 전문가·취미가가 인터넷 연결 기기를 발견·분석하도록 돕는 검색엔진.

### 어떻게 동작하나 (HTTP 헤더 공부와 연결)
- 구글은 페이지 **본문(body)**을 인덱싱 / Shodan은 **HTTP 헤더 + 배너(banner)** 정보를 인덱싱.
- 포트를 스캔해서 서버가 응답으로 뱉는 배너(서비스 종류, 버전, OS 등)를 수집.
- 대상 프로토콜: HTTP, HTTPS, FTP, SSH, Telnet, SNMP, SIP 등. 지역·필터로 검색 가능.
- 즉 **배너 그래빙(banner grabbing)을 인터넷 전체 규모로 자동화**한 것.

### "그러면 폐쇄해야 하지 않나?" → 폐쇄해도 의미 없고, 오히려 방어에 유용

1. **Shodan은 취약점을 만들지 않는다** — 이미 열려 노출된 기기를 목록화할 뿐. Shodan이 없어도 그 기기는 열려 있음. 진짜 공격자는 `masscan`, `zmap`으로 직접 스캔(흔적 안 남김). 막아도 노출 기기는 그대로.
2. **방어 쪽에서 더 많이 쓴다** — 주 사용자는 침투 테스터, 법 집행기관, 학계, 보안 인력. 사이버보험사(패치 여부 평가), M&A 보안 실사, 제조사(패치 안 된 자사 SW 탐색) 등이 활용.
3. **노출 사실을 드러내는 게 보안에 이롭다** — Matherly 철학: "성벽에만 의존할 수 없다. 거대한 성을 지어도 이웃이 안전하지 않으면 소용없다." 노출이 가려지면 관리자도 자기가 뚫린 줄 모름.

### 비유
Shodan = "이 동네 어느 집 현관문이 열려 있다"를 알려주는 **지도**.
- 문을 연 건 집주인(노출된 기기), 도둑은 지도 없어도 골목 돌며 찾음.
- 지도가 공개돼야 집주인·보안팀도 "어, 문 열렸네" 하고 닫음. 지도를 없앤다고 문이 닫히진 않음.

> 핵심 논쟁: **투명성을 통한 보안 vs 은폐를 통한 보안(security through obscurity)**.
> 업계 다수 의견 = "숨긴다고 안전해지지 않는다." Shodan은 그 철학의 산물.

### 윤리·법적 선 (중요)
- **목록에서 "보는 것"** ≠ **실제 접속/로그인하는 것**.
- 후자는 한국 정보통신망법상 명백한 불법(무단 침입).
- 연구·학습 목적이면 **"관찰과 통계"까지만** 안전.
