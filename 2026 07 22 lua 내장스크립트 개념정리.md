# Lua 내장 스크립트 개념 정리

작성일: 2026-07-22

---

## 1. Lua vs XSLT — 둘 다 "내장 스크립트"인가

### Lua
- 범용(general-purpose) 언어. 변수·반복문·함수·조건문을 갖춘 명령형(imperative) 언어임.
- 처음부터 "호스트 프로그램에 끼워 넣기"를 목적으로 설계됨. 인터프리터가 가볍고 C API가 작아 통합이 쉬움.
- 게임 엔진, Redis, nginx(OpenResty), Wireshark 등에 널리 내장됨.
- 전형적인 임베디드 스크립팅 언어에 해당함.

### XSLT
- 범용 언어가 아니라 XML 변환 전용 도메인 특화(domain-specific) 언어임.
- 명령형이 아니라 선언형(declarative)에 가까움. "특정 패턴의 노드를 만나면 이렇게 변환하라"는 템플릿 매칭 방식.
- 엄밀히 "임베디드 스크립팅 언어"로 분류되지 않음. "변환 규칙(stylesheet)을 외부 파일로 정의"하는 쪽에 가까움.

### S-100 Portrayal에서 둘 다 쓰는 이유
- Portrayal catalogue는 피처를 어떻게 그릴지(심볼·색·선 스타일) 정하는 규칙 집합임. 이 규칙 표현 수단으로 XSLT 또는 Lua를 채택함.
- XSLT: S-100 데이터가 XML/GML 기반이라 XML 변환에 자연스러움. 단, 조건·계산이 복잡해지면 장황해짐.
- Lua: 복잡한 분기·계산 로직을 절차적으로 간결하게 작성 가능함.
- 즉 IHO가 구현자에게 "선언적 XML 변환이 편하면 XSLT, 복잡한 절차적 로직이 편하면 Lua"로 선택지를 제공한 구조임.
- 요약: "규칙을 외부로 분리해 유연하게 바꾼다"는 목적은 유사하나, "임베디드 스크립팅 언어" 타이틀에 맞는 것은 Lua이고 XSLT는 결이 다른 도구임.

## 2. Lua는 단독으로도 실행되는가

- 단독 실행 가능함. Lua는 독립 인터프리터(standalone interpreter)인 `lua` 실행 파일을 제공함.

```bash
lua script.lua        # 파일 실행

lua                   # 대화형 모드(REPL)
> print("hello")
hello
```

- 단, `lua` 인터프리터 자체도 사실은 Lua 코어를 호스트한 작은 C 프로그램임. Lua는 원래 라이브러리(`liblua`)로 배포되고, 그것을 감싼 최소한의 호스트가 `lua` 커맨드라인 툴임. 따라서 "임베디드 언어인데 단독 실행된다"는 모순이 아니라, 얇은 기본 호스트가 딸려 오는 구조임.

### 단독 실행 vs 임베디드 차이
- 단독 실행: `lua` 커맨드가 표준 라이브러리(`print`, `io`, `os`, `string`, `math` 등)를 붙여줘 일반 스크립트 언어처럼 파일 실행·REPL이 가능함.
- 임베디드: 게임 엔진·nginx 같은 호스트가 자기 함수를 Lua에 주입함. Lua 코드는 호스트가 노출한 API만 사용 가능하고, 표준 라이브러리 일부는 보안상 차단되기도 함.
- 같은 Lua 문법이라도 무엇을 호출할 수 있느냐가 호스트에 따라 달라짐. S-100 Portrayal의 Lua도 Portrayal 엔진이 노출한 드로잉 관련 함수·데이터에 접근하는 형태라 순수 단독 실행 Lua와 사용 가능 API가 다름.

## 3. Lua 실행에 C가 필요한가

- Lua를 "실행"만 할 때는 C 불필요함. 미리 빌드된 `lua` 실행 파일(바이너리)만 있으면 됨. 바이너리는 누군가 C 컴파일러로 미리 빌드해둔 결과물이므로 받아서 쓰면 됨.

```bash
apt install lua5.4     # 우분투/데비안
brew install lua       # macOS
lua script.lua
```

### C가 필요한 경우
1. Lua 인터프리터 자체를 소스에서 직접 빌드할 때. Lua는 C로 작성돼 있어 소스(`lua-5.4.x.tar.gz`)를 직접 컴파일하려면 C 컴파일러 필요함. 패키지 매니저로 미리 빌드된 것을 받으면 이 과정 생략됨.
2. Lua를 다른 C 프로그램에 임베딩할 때. 호스트 프로그램을 C로 작성·컴파일해야 함.

| 하려는 것 | C 필요? |
|---|---|
| 이미 있는 `lua`로 스크립트 실행 | ❌ |
| Lua를 소스에서 직접 빌드 | ✅ (한 번만) |
| 내 C 프로그램에 Lua 임베딩 | ✅ |

- "Lua가 C로 만들어졌다"는 사실과 "Lua를 쓰려면 C가 있어야 한다"는 별개임. 만들어진 바이너리를 쓰는 입장에선 C 불필요함.
- S-100 Portrayal의 Lua는 위 세 경우 어디에도 해당하지 않음. Portrayal 엔진(뷰어/구현 소프트웨어)이 내부에 Lua를 이미 품고 있어, 사용자는 Lua 스크립트만 작성하면 엔진이 실행해줌. 엔진 제작자는 Lua를 임베딩했지만, 스크립트 작성자는 C를 신경 쓸 필요 없음.

## 4. 언어와 구현체 구분 — Python도 C로 만들어졌는가

- 우리가 보통 쓰는 Python은 정확히는 **CPython**임. C로 작성된 구현체이며 사실상 표준으로 취급됨.
- 핵심 구분: "언어"와 "구현체"는 다름.
  - 언어: 문법·규칙의 명세(스펙). 특정 프로그래밍 언어로 만들어진 것이 아님.
  - 구현체: 그 명세를 실제로 돌아가게 만든 프로그램. 무언가로 만들어야 함. CPython은 이를 C로 만든 것임.
  - 같은 Python 언어를 다른 언어로 구현한 것도 있음: Jython(자바), PyPy, IronPython(C#) 등.
- Lua도 동일 구조임. 표준 Lua는 C로 구현됨(PUC-Rio Lua). LuaJIT처럼 다른 방식의 구현체도 있음.

### Lua는 고급(고수준) 언어인가
- "고급 언어"는 두 가지 의미로 쓰임.
- 저수준/고수준(low-level/high-level) 축: 하드웨어에서 얼마나 멀리 떨어진 추상화냐의 기준.
  - C: 메모리 주소를 직접 다루는 저수준.
  - Lua·Python: 자동 메모리 관리(GC), 읽기 쉬운 문법을 쓰는 고수준. 이 의미에서 Lua는 고수준 언어 맞음.
- 주의할 인과관계: "C로 만들어졌기 때문에 고급"이 아님.
  - 어떤 언어를 C로 구현했어도 그 언어가 여전히 저수준일 수 있음. "무엇으로 만들어졌는가(구현 언어)"와 "그 언어가 고수준이냐(추상화 수준)"는 별개 축임.
  - Lua가 고수준인 이유는 C로 만들어져서가 아니라, Lua 언어 자체가 제공하는 추상화가 높기 때문임(자동 메모리 관리, 동적 타입, 읽기 쉬운 문법).

## 5. Lua가 가벼운 이유

1. 코어 구현 자체가 작음. 인터프리터 전체가 C 소스 몇만 줄 수준, 컴파일 시 바이너리 수백 KB 수준. 대형 언어 런타임(수십 MB)과 자릿수가 다름.
2. 표준 라이브러리를 최소한만 둠. 문자열·테이블·수학·io·os 정도. 필요한 기능은 호스트가 주입하거나 사용자가 붙이는 방식이라 코어가 얇게 유지됨.
3. 의존성이 거의 없음. 순수 ISO C(C89)만으로 작성돼 표준 C 컴파일러만 있으면 어디서든 빌드됨. 이식성이 높음.
4. 자료구조를 하나로 통일함. **테이블(table)** 단일 자료구조로 배열·딕셔너리·객체·네임스페이스를 모두 처리함. 언어가 제공할 기본 요소 수가 줄어 구현이 단순·경량화됨.
5. 런타임 메커니즘이 단순함. 바이트코드로 컴파일해 레지스터 기반 VM에서 실행함. 구조가 단순하고 초기화 부담이 적어 기동이 빠름.

- Lua의 가벼움은 설계 철학의 결과임. "작고, 의존성 없고, 남의 프로그램에 끼워 넣기 좋게"를 처음부터 목표로 설정함. 기능을 코어 밖으로 밀어내고(호스트가 채우게 하고) 언어 요소를 최소화함.
- 이 가벼움 덕에 S-100 Portrayal 엔진 등에 부담 없이 내장 가능함.

## 6. Lua로 TCP 통신

- Lua 단독으로는 불가함. 외부 라이브러리 필요함.
- Lua 표준 라이브러리에는 네트워크 기능이 없음(`string`, `table`, `math`, `io`, `os` 정도). 코어를 얇게 유지하는 철학의 결과임.

### LuaSocket (사실상 표준)

```bash
luarocks install luasocket
```

- `luarocks`는 Lua의 패키지 매니저임(Python pip에 대응).

TCP 클라이언트 예시:

```lua
local socket = require("socket")

local client = socket.connect("example.com", 80)
client:send("GET / HTTP/1.0\r\n\r\n")

local response = client:receive("*a")  -- 전부 받기
print(response)

client:close()
```

TCP 서버 예시:

```lua
local socket = require("socket")

local server = socket.bind("*", 8080)  -- 8080 포트 대기

while true do
  local client = server:accept()       -- 연결 수락
  local line = client:receive()        -- 한 줄 받기
  client:send("got: " .. line .. "\n")
  client:close()
end
```

### 선택지
- **LuaSocket**: 가장 범용적. 동기(blocking) 방식 기본.
- **cqueues**: 코루틴 기반 비동기 네트워킹. 다중 연결 동시 처리에 강함.
- **호스트 제공 API**: Lua가 임베딩된 상태면 호스트가 자기 네트워크 함수를 노출할 수 있음. OpenResty(nginx+Lua)는 자체 비동기 소켓 API(`ngx.socket.tcp`)를 제공하며, nginx 이벤트 루프에 얹혀 도는 방식이라 효율적임.

### 임베디드 맥락 주의점
- 임베딩된 Lua에서는 네트워크가 차단돼 있을 수 있음. 호스트가 보안·용도상 소켓 라이브러리를 붙이지 않으면 TCP 통신 불가함.
- S-100 Portrayal 스크립트는 "데이터를 어떻게 그릴까"만 계산하는 용도라 네트워크 통신 기능이 노출되지 않는 것이 정상임.
- 결론: 범용 Lua는 LuaSocket으로 가능, 임베딩된 Lua는 호스트가 허용해야 가능함.

## 7. Lua의 메모리 사용

- "메모리를 적게 쓴다"를 두 축으로 구분함.

### 고정 비용 — 런타임 자체가 차지하는 메모리
- 인터프리터를 하나 띄웠을 때 기본 점유 메모리가 매우 적음. 코어가 작고 표준 라이브러리도 최소라, 프로그램에 Lua를 심어도 수십~수백 KB 수준임. 임베딩 시 부담이 거의 없는 이유임.

### 동적 비용 — 실행 중 데이터 처리 메모리
- 값의 종류가 적고(테이블 단일화 기여) 내부 구조가 단순해, 객체 하나에 붙는 부가 정보가 적음.

### "적게 쓴다" ≠ "관리 안 한다"
- Lua도 데이터를 만들면 메모리를 씀. 다만 **가비지 컬렉터(GC)** 가 안 쓰는 메모리를 자동 회수함. 사람이 직접 비울 필요 없음(고수준 언어 특성). C였다면 `free`로 직접 해제해야 하는 부분임.
- 정리:
  - 적게 쓰냐 → 코어도 데이터 처리도 알뜰한 편.
  - 알아서 치우냐 → GC가 자동 회수함.

### 균형
- 절대적으로 "거의 안 쓴다"는 아님. 데이터를 많이 만들면 그만큼 씀. 포인트는 같은 작업 시 다른 런타임보다 오버헤드가 작다는 것이며, 특히 "런타임을 심는 비용이 싸다"는 점이 임베디드 언어로서 최대 강점임.

## 8. Lua를 임베딩한 대표 사례 + 호스트-내장 스크립트 패턴

### Lua를 내장한 시스템

| 호스트 | Lua 용도 |
|---|---|
| **Redis** | 서버 내부에서 원자적으로 도는 스크립트(`EVAL` 명령). 여러 연산을 한 번에 처리 |
| **Nginx (OpenResty)** | 요청 처리 로직을 Lua로 작성 (`ngx.socket` 등) |
| **Wireshark** | 패킷 분석용 커스텀 디섹터(dissector)를 Lua로 작성 |
| **Neovim** | 에디터 설정·플러그인을 Lua로 (Vimscript 대체) |
| **게임 엔진** | Roblox, WoW 애드온, 다수 게임의 모드/퀘스트 로직 |
| **HAProxy** | 로드밸런서 동작을 Lua로 커스터마이징 |

- Redis가 대표 사례임. C로 작성된 초경량 인메모리 서버에 무거운 스크립트 엔진을 붙이면 "가볍고 빠름"이라는 존재 이유가 무너지므로, 심어도 가볍고 원자적 실행 보장이 쉬운 Lua를 채택함.

### 같은 "호스트 + 내장 스크립트" 패턴, 다른 언어

- **QGIS, Blender, Autodesk Maya** → Python 내장
- **Excel, 여러 웹 브라우저** → JavaScript 계열 내장
- **Emacs** → Emacs Lisp 내장
- **VS Code** → JavaScript/TypeScript로 확장

### 핵심 구조 대응
- **S-100 Portrayal의 Lua ≈ QGIS의 Python**. 호스트가 언어를 품고, 자기 기능을 API로 노출하고(예: QGIS의 `qgis.core`), 스크립트는 그 노출된 API로 호스트를 제어·확장함. 언어만 다를 뿐 패턴이 동일함.
- 큰 그림: "복잡한 핵심 기능은 C/C++로 빠르게 작성하고, 사용자 커스터마이징 부분은 가벼운 스크립트 언어로 노출한다"는 소프트웨어 설계의 흔한 전략임. Lua는 그중 "최대한 안 무겁게 심고 싶을 때" 특히 강한 선택지임.
- S-100 Portrayal이 Lua/XSLT를 고른 것도 동일 맥락임. Portrayal 규칙을 뷰어마다 심어야 하는데 무거운 런타임을 강제하면 구현 부담이 커지므로 가벼운 것을 선택함.
