# MQTT 브로커 개념 정리 (2026-06-02)

MQTT 브로커가 뭔지, 왜 회사가 제품으로 만들 정도인지, 실제 운영은 어떻게 되는지 정리.

---

## 1. HiveMQ — MQTT 브로커 만드는 회사

- 독일 회사. **MQTT 브로커**를 엔터프라이즈급으로 만들어 파는 곳.
- 브로커 = IoT 디바이스들이 publish한 메시지를 받아서 subscribe한 쪽으로 전달하는 중간 다리(서버).
- 강점: 대규모 IoT(수백만 동시 연결), 커넥티드카·제조·물류 분야, MQTT 5.0 같은 최신 스펙 빠른 지원, 클러스터링·고가용성·보안·모니터링 같은 운영 기능.
- 무료로 접할 수 있는 것:
  - HiveMQ Community Edition (오픈소스)
  - HiveMQ Public Broker (`broker.hivemq.com`) — 공개 테스트용
  - HiveMQ MQTT Client (Java용 클라이언트 라이브러리)
- 경쟁/대안: Eclipse Mosquitto(가벼움), EMQX(대규모 강점).

---

## 2. 브로커 개념은 작다, 근데 "구현물"은 거대하다

핵심 포인트: **개념의 크기 ≠ 제품의 크기**

- MQTT 스펙이 정의하는 브로커는 단순함 → "토픽으로 발행하면 그 토픽 구독자에게 전달해라"가 본질의 전부. 학습용으로는 며칠이면 동작하는 걸 만들 수도 있음.
- 회사 하나를 먹여 살리는 건 그 단순한 동작을 **현실에서 진짜로 돌릴 때** 생기는 모든 것:
  - **규모** — 10개 연결과 1,000만 동시 연결은 완전히 다른 문제(메모리·동시성·커넥션 풀)
  - **장애 대응** — 클러스터링으로 서버 하나 죽어도 메시지 안 끊기게 (분산 시스템의 정수)
  - **메시지 보장** — QoS 0/1/2, "정확히 한 번" 보장하려면 상태 저장 + 재전송 로직
  - **보안·인증·모니터링·운영** — TLS, 권한 관리, 대시보드, 로깅… 본체보다 방대

비유:
- HTTP도 단순한 프로토콜이지만 구현물인 nginx/Apache/CDN은 거대함.
- DB도 "저장하고 꺼낸다"는 한 줄 개념이지만 PostgreSQL/Oracle은 초거대 소프트웨어.
- 프로토콜 스펙은 **무엇을 할지(behavior)**만 정의, **어떻게 깨지지 않게 대규모로 구현하느냐(implementation)**는 회사들의 몫.

→ "브로커가 규모가 큰 개념"이 아니라, **단순한 개념을 현실에서 감당 가능하게 만드는 엔지니어링이 거대한 것.**

## 3. Mosquitto — 가볍고 단순한 오픈소스 브로커

- 정식명 Eclipse Mosquitto. **브로커 맞음.** HiveMQ와 같은 카테고리.
- **오픈소스 맞음.** Eclipse Foundation 산하, EPL/EDL 듀얼 라이선스, 무료.
- 왜 많이 쓰이나 = "가볍고 단순"에 특화:
  - C로 작성, 리소스 거의 안 먹음 → 라즈베리파이·임베디드에서도 동작
  - `apt install mosquitto` 한 줄이면 끝, 학습용 최적
  - 단일 프로세스로 소~중규모는 충분
- 딸려오는 CLI 도구: `mosquitto_pub`, `mosquitto_sub` → 터미널에서 바로 발행/구독 테스트 가능

**포지션 차이**
- Mosquitto → 가볍고 단순, 학습·소규모·엣지, 단일 서버 중심
- HiveMQ / EMQX → 무겁지만 대규모, 클러스터링·HA·엔터프라이즈, 수백만 연결
- 보통 공부·프로토타입은 Mosquitto, 대규모 프로덕션 가면 HiveMQ 검토

## 4. "Mosquitto = Flask?" → 카테고리가 다름

직관(가볍고+인기+오픈소스)은 맞지만, **물건의 종류**가 어긋남.

- **Flask = 프레임워크/라이브러리** → 내가 코드를 써서 *내 서버를 만드는* 재료
- **Mosquitto = 완성된 실행 프로그램(서버 데몬)** → 설치하고 실행하면 바로 동작, 코드 안 짬

→ 그래서 Mosquitto는 Flask보다 **nginx/Apache에 가까움.** "MQTT 세계의 nginx" 느낌.

**그럼 Flask에 대응하는 건?**
- MQTT **클라이언트 라이브러리** (파이썬 `paho-mqtt`, Java면 HiveMQ MQTT Client). 내 코드에 넣어서 pub/sub 로직을 *내가 짜는* 것.

정리:
| 종류 | MQTT | 웹 |
|------|------|-----|
| 완성품 서버 | 브로커 (Mosquitto, HiveMQ) | nginx, Apache |
| 코드에 넣는 재료 | 클라이언트 라이브러리 (paho-mqtt) | Flask, requests |

다만 **포지셔닝 축**("가벼움 vs 풀패키지")은 정확히 맞음: Flask↔Mosquitto, Django↔HiveMQ 느낌.

## 5. 브로커는 24시간 상시 떠 있어야 함

- MQTT는 발행자와 구독자가 **직접 연결 안 됨.** 모든 메시지가 무조건 브로커를 거침.
- 즉 **브로커가 꺼지면 발행도 구독도 다 멈춤** (발행자·구독자가 살아있어도 소용없음).
- 예: BlueMap 서버(발행) → "수온 18도" publish → 브로커 → 선박/앱(구독) subscribe

**운영 선택지 2가지**
1. **직접 운영 (self-hosted)** — EC2에 Mosquitto/HiveMQ 직접 설치하고 상시 운영. 기존 weather-api/SECOM 올린 방식 그대로. 통제권 완전 확보, 대신 모니터링·관리 책임도 내가 짐.
2. **매니지드 (managed)** — AWS IoT Core, HiveMQ Cloud 등. 운영 부담 없이 엔드포인트에 연결만. 대신 비용 + 통제권 일부 양보.

→ 지금 EC2 기반 흐름이면 처음엔 **EC2에 Mosquitto 컨테이너 하나 추가**가 자연스러움 (FastAPI+Docker+cron 아키텍처에 붙이기 쉬움).

**생각할 거리:** 기상데이터에 MQTT가 맞는지는 데이터 성격에 따라 갈림. MQTT는 "값이 자주 바뀌고 변할 때마다 실시간 push" 시나리오에서 빛남. 주기적으로 받아가는 형태라면 다른 방식이 나을 수도.

## 6. 사양 — 1GB 컨테이너로 충분하다

- Mosquitto는 idle 상태에서 **몇 MB 수준**. 라즈베리파이에서도 돌아감. 1GB면 넉넉.
- 메모리를 결정하는 건 "브로커 자체"가 아니라 **연결 수 × 메시지 적체량**:
  - **동시 연결 수** — 수백~수천까진 1GB로 가뿐. 수만~수십만 가면 얘기 달라짐.
  - **QoS와 메시지 적체** — QoS 1/2는 전달 확인까지 메시지 보유, 구독자 끊긴 동안 쌓이는 queued 메시지가 진짜 변수.
  - **retained 메시지 / persistent 세션** — 마지막 값 보유, 끊긴 세션 유지 시 추가 사용.

→ BlueMap 기상데이터 규모(발행자 몇 개, 구독자 수십~수백)면 **1GB로 전혀 무리 없음, 여유 많음.**

**1GB 안에서 챙길 안전장치(가드레일)**
- `max_queued_messages` — 구독자 끊겼을 때 무한정 쌓이는 것 방지 (메모리 슬금슬금 차오름 차단)
- `max_connections` — 연결 수 상한, 폭주 보호
- `message_size_limit` — 페이로드 크기 제한 (기상데이터는 작은 JSON이라 보통 문제없음)
- 컨테이너 메모리 하드리밋 — Docker `--memory=1g`로 호스트 전체 번짐 방지

→ 사양 걱정 X. 신경 쓸 건 사양보다 **큐·연결 상한 설정**으로 메모리를 예측 가능하게 유지하는 것.
